# ITエンジニア市場データELTプロジェクト 作業メモ

## 目的

本プロジェクトは、ITエンジニア職種に関する検索関心データと求人需要データを収集し、SQLを中心に `raw → staging → mart` の流れで整形・集計するためのポートフォリオ作成用メモです。

公開用READMEではなく、作業工程・判断事項・タスクを整理するための内部メモとして使用します。

## プロジェクトの主目的

本プロジェクトの主目的は、分析結果を深掘りすることではなく、以下の工程を見せることです。

- 外部データを収集する
- rawデータとして保存する
- DBに取り込む
- SQLでstaging層に整形する
- SQLでmart層を作成する
- データ品質チェックを行う
- 可視化用データを作成する

分析・考察は、データマートの利用例として最小限に扱います。

## 初期版の対象範囲

### 対象期間

- 2015年1月〜2025年12月

### 対象期間を2015〜2025年にした理由

当初は、e-Stat側の現行分類に合わせて `2022年4月〜2026年3月` を初期対象にする案を検討しました。

しかし、Google Trendsで職種名の検索関心を扱う場合、AI・データ・インフラ系職種への関心変化は数年単位では見えにくく、より長期の推移を見る方がテーマに合うと判断しました。

また、e-Stat / 厚生労働省「職業安定業務統計」についても、2015年まで遡れるデータを取得できたため、初期版から `2015年〜2025年` の長期データで進めます。

### Google Trendsの取得期間を分割しない理由

Google Trendsの値は、実検索回数ではなく、指定した条件内で0〜100に正規化された検索関心指数です。

そのため、以下のように期間を分割して取得すると、各期間ごとに正規化基準が変わります。

```text
2015〜2020年
2021〜2025年
```

この場合、2015〜2020年の「100」と、2021〜2025年の「100」は同じ意味になりません。

そのため、本プロジェクトでは比較対象期間を固定し、Google Trendsは以下の条件で一括取得します。

```text
対象期間：2015/01/01〜2025/12/31
地域：日本
検索種別：ウェブ検索
カテゴリ：すべてのカテゴリ
```

## 初期版で扱う分析・可視化

### 1. e-Stat / 厚生労働省 職業安定業務統計による求人需要推移

対象期間は、Google Trendsと揃えて `2015年1月〜2025年12月` とします。

初期版では、取得済みの第21表をもとに、IT系職種に関連する求人需要指標を月次で抽出・整形します。

使用ファイルは以下です。

```text
一般職業紹介状況（職業安定業務統計）2012-2022.xlsx
一般職業紹介状況（職業安定業務統計）2023-2025.xlsx
```

扱う職業分類は、初期版では以下に絞ります。

```text
情報処理・通信技術者
```

扱う指標は以下です。

- 新規求人
- 有効求人
- 有効求人倍率

第21表には左側に年次データ、右側に月次データが含まれています。Google Trendsも月次データとして取得しているため、e-Stat側も月次列を抽出して `staging` 層に保持します。

#### e-Statファイル間の重複データの扱い

`2012-2022` ファイルと `2023-2025` ファイルには、2022年データが重複して含まれています。

確認したところ、重複している2022年の数値は同じだったため、重複取込を避けるために以下のルールで採用します。

```text
2012-2022ファイル：2015年1月〜2022年12月を採用
2023-2025ファイル：2023年1月〜2025年12月を採用
```

つまり、`2023-2025` ファイル側に含まれる2022年データは使用しません。

### 2. Google Trendsによる職種別の検索関心推移

Google Trendsでは、以下の7KWを対象にします。

```text
システムエンジニア
ITエンジニア
プログラマー
Webエンジニア
インフラエンジニア
データエンジニア
AIエンジニア
```

この7KWを使い、2015年〜2025年における職種名の検索関心推移を確認します。

### 3. 検索関心と求人需要の比較用データマート作成

Google Trendsとe-Statは、データの意味・粒度・分類が異なります。

そのため、両者を直接的な因果関係として扱うのではなく、月次データとして整形し、比較可能なmartを作成します。

データ保持・加工の粒度は月次とし、長期推移の基本可視化では年次集計を使用します。

- 月次の検索関心指数
- 月次の求人需要指標
- 年次集計した検索関心指数
- 年次集計した求人需要指標
- データソース別・指標別の比較用テーブル

月次データは、特定年や特定イベント前後の変化を確認する場合に使用します。

## データ粒度と比較粒度の方針

本プロジェクトでは、Google Trendsとe-Statの両方で `2015年1月〜2025年12月` の月次データを取得できています。

そのため、`raw` / `staging` 層では月次データとして保持します。

一方で、2015年〜2025年の長期傾向を確認する場合、月次のまま可視化すると変動が細かくなりすぎるため、基本の比較・可視化では `mart` 層で年次集計したデータを使用します。

```text
raw：取得元ファイルをそのまま保存
staging：月次・縦持ちデータに整形
monthly mart：月次比較・詳細確認用
yearly mart：長期推移の基本可視化用
```

### 年次集計の考え方

Google Trendsは、月次の検索関心指数を年平均に集計します。

e-Statの求人需要指標も、初期版ではGoogle Trendsとの比較・可視化のしやすさを優先し、月次値の年平均を使用します。

ただし、新規求人は本来、年間累計として見る方法もあります。初期版では年平均で統一し、必要に応じて拡張時に年間累計の指標を追加します。

## Google TrendsのKW選定

### 最終的に採用するKW

| KW | 役割 |
|---|---|
| システムエンジニア | 日本の従来型IT職種名。アンカーKWとしても使用 |
| ITエンジニア | IT職種全体を表す総称 |
| プログラマー | 従来型の開発職種名 |
| Webエンジニア | Web開発系の代表KW |
| インフラエンジニア | ネットワーク・サーバー・クラウドを含む基盤系の代表KW |
| データエンジニア | DB・DWH・ELT・データ基盤系の代表KW |
| AIエンジニア | AI・機械学習系の代表KW |

### KWを絞った理由

当初は、より多くのエンジニア職種名を比較対象にする案を検討しました。

候補には以下がありました。

```text
ソフトウェアエンジニア
フロントエンドエンジニア
バックエンドエンジニア
ネットワークエンジニア
クラウドエンジニア
サーバーエンジニア
セキュリティエンジニア
データベースエンジニア
機械学習エンジニア
```

ただし、職種名を細かく分けすぎると、以下の問題があります。

- Google Trends上で検索ボリュームが小さくなり、0や低い値が増えやすい
- 似た職種同士の重複が増え、比較軸が分かりにくくなる
- ポートフォリオの主目的であるELT構築よりも、職種分類分析に寄りすぎる

そのため、初期版では細かい職種名を代表KWに統合します。

### 除外・統合したKW

| 除外・統合したKW | 初期版での扱い | 理由 |
|---|---|---|
| ソフトウェアエンジニア | 除外 | プログラマー・Webエンジニアと近く、今回は開発系代表として整理 |
| フロントエンドエンジニア | Webエンジニアに統合 | Web開発系の細分化KWとして扱う |
| バックエンドエンジニア | Webエンジニアに統合 | Web開発系の細分化KWとして扱う |
| ネットワークエンジニア | インフラエンジニアに統合 | 基盤系職種の一部として扱う |
| クラウドエンジニア | インフラエンジニアに統合 | 基盤系職種の一部として扱う |
| サーバーエンジニア | インフラエンジニアに統合 | 基盤系職種の一部として扱う |
| セキュリティエンジニア | 除外 | 初期版ではDB・データ基盤寄りのPF目的から優先度を下げる |
| データベースエンジニア | データエンジニアに統合 | 検索ボリュームが小さく、データ基盤系の代表としてデータエンジニアを採用 |
| 機械学習エンジニア | AIエンジニアに統合 | 検索KWとしてはAIエンジニアの方が一般的で見えやすいと判断 |

## Google Trendsの取得グループ

Google Trendsは1回に比較できるKW数に上限があり、また取得結果はそのグループ内で0〜100に正規化されます。

そのため、複数グループに分けて取得する場合は、共通のアンカーKWを含めます。

### アンカーKW

```text
システムエンジニア
```

### アンカーKWを置いた理由

Google Trendsでは、異なるKWグループを別々に取得すると、それぞれのグループ内で正規化されます。

そのため、以下のような2つのCSVをそのまま比較すると、基準がずれます。

```text
Group A：システムエンジニア、ITエンジニア、プログラマー、Webエンジニア
Group B：システムエンジニア、インフラエンジニア、データエンジニア、AIエンジニア
```

この問題を避けるため、両方のグループに共通して含まれる `システムエンジニア` をアンカーKWとして使用します。

`システムエンジニア` は、日本のIT職種名として一般的で、検索量も比較的安定しているため、アンカーに適していると判断しました。

### 取得グループ

#### Group 1：総称・開発系

取得ファイル：

```text
システム、IT、プログラマー、Web.csv
```

KW：

```text
システムエンジニア
ITエンジニア
プログラマー
Webエンジニア
```

#### Group 2：基盤・データ・AI系

取得ファイル：

```text
システム、インフラ、データ、AI.csv
```

KW：

```text
システムエンジニア
インフラエンジニア
データエンジニア
AIエンジニア
```

### グループ間補正の方針

Group 1とGroup 2は、それぞれ別々に0〜100へ正規化されています。

そのため、最終的に7KWを同じデータマートで比較する場合は、アンカーKWである `システムエンジニア` を使って補正します。

初期版では、以下の簡易補正を検討します。

```text
補正係数 = Group 1のシステムエンジニア平均値 ÷ Group 2のシステムエンジニア平均値
Group 2補正後値 = Group 2元値 × 補正係数
```

補正後の値も実検索数ではなく、あくまで比較用に補正した検索関心指数として扱います。

## 注意点

- Google Trendsの値は実検索回数ではなく、検索関心指数として扱う
- Google Trendsは取得期間・取得グループによって0〜100の基準が変わる
- Google Trendsを複数グループで取得する場合は、アンカーKWを使って補正する
- 補正後の値も実検索数ではなく、比較用の補正済み検索関心指数として扱う
- e-Statの職業分類と検索KWの職種名は完全には一致しない
- e-Statは2012-2022ファイルと2023-2025ファイルを連結して使う
- 2022年は両ファイルに重複して含まれるため、2012-2022ファイル側を採用し、2023-2025ファイル側の2022年データは除外する
- raw / stagingでは月次データを保持し、基本の比較・可視化では年次集計を使う
- 求人需要と検索関心の関係は因果ではなく、傾向比較として扱う
- 分析の深掘りより、データ取得・整形・集計工程の再現性を重視する
